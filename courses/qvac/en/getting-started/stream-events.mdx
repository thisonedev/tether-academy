In the previous lesson we used `result.tokenStream`, which is a flat string iterable. That works for "drain tokens into a UI", but as soon as you want to know *what kind* of token it is, you need the richer event stream.

`completion()` returns two surfaces: `tokenStream` (the simple string iterable) and `events` (a typed `AsyncIterable<CompletionEvent>`). The events stream gives you distinguishable events for content deltas, thinking deltas, tool calls, and a final `completionDone` with stats.

Let's set `captureThinking: true` so we can see both content and thinking events fire side by side:

```ts
const result = completion({
  modelId,
  history: [{ role: "user", content: "Explain quantum computing in 2 sentences" }],
  stream: true,
  captureThinking: true,
});
```

If you want to run a loop over the events, you can use `for await` and a `switch` statement to dispatch on `event.type`:

```ts
for await (const event of result.events) {
  switch (event.type) {
    case "contentDelta":
      process.stdout.write(event.text);
      break;
    case "thinkingDelta":
      process.stderr.write(`[think] ${event.text}`);
      break;
    case "completionStats":
      if (event.stats.tokensPerSecond !== undefined) {
        console.log(`\n▸ ${event.stats.tokensPerSecond.toFixed(0)} tok/s`);
      }
      break;
  }
}
```

In this case, `contentDelta` is the model's text token-by-token, which goes to stdout. `thinkingDelta` is the chain-of-thought stream, which goes to stderr so it doesn't mix with the response. `completionStats` carries throughput numbers like tokens-per-second.

> Note: `tokenStream` is still around for simple cases, but new code should consume `events` for streaming and `final.contentText` for the aggregated result.

## Put it to the test

1. Call `completion({ ...captureThinking: true, stream: true })`.
2. Iterate `result.events`. On `contentDelta`, write `event.text` to stdout. On `thinkingDelta`, write `[think] ${event.text}` to stderr. On `completionStats`, log the `tokensPerSecond` rate.
